## Standard checklist

- **Critical-path integrity** ✅ — every ambient adapter is a background worker job → control-plane `/ingest`. None sit on `button → ESP32 feedback → local UI`. No door-Pi service imports any ambient integration (verified: zero references to the integration packages or their credentials in `door-api`/`door-visiond`/`door-media`/`door-sync`).
- **Privacy (ARCHITECTURE.md §9)** ✅ — ambient payloads carry no PII/biometrics (aircraft: callsign/alt/dist/heading only, explicitly tested to exclude route/origin/destination; food: meal names; printer: state/job/progress/eta, camera stream URL deliberately kept out of the payload). Photo booth is a **feature-gated private** gallery with consent metadata, tokenized routes, and wired deletion paths — no private media on public routes.
- **Trust boundaries** ✅ — integration credentials (OpenSky user/pass, OctoPrint API key) live only in `wallboard-worker` settings (NUC), never on the Pi. Public/visitor endpoints remain signed-token + rate-limited. ESP32 stays secret-free.
- **Contract discipline** ✅ — all six use `packages/contracts` events; UTC `occurred_at` + `monotonic_ms` conventions held; `check-generated-contracts` green.

- **Tests / latency** ✅ — full `pytest` green; each worker adapter has a job-level ingest test + degradation test; the T-702 sim soak (`tests/performance/soak.py`) passes on this `main` with no budget regressions. (Physical glass-to-glass p95 is a sim-execution proxy only — see risks.)
- **Drift** — the README status line was stale at "M0 complete"; updated to reflect M6 in this PR. Future-task brief `T-701-security-review.md` refined with the ambient egress inventory below.
- **ADRs** — no new architectural decisions emerged; all six adapters followed the established M6 pattern. No ADR needed.

## M6-specific checks (per T-x90 template)

- **Each ambient tile degrades independently** ✅ — every job isolates failure: BirdNET/satellite/aircraft return `None` (tile goes stale), printer posts an explicit `offline` event, food falls back to yesterday's cached recommendation then goes stale silently. A failure in one adapter cannot affect another (separate worker commands/jobs) or the door path.
- **Feature flags verified** ✅ — `FEATURE_BIRDNET/SATELLITES/AIRCRAFT/PRINTER/FOOD` all default `False`; the CLI selects the mock provider whenever the flag is off, so a fresh deploy runs safely in mock mode until explicitly configured.
- **No integration credential on the Pi** ✅ — confirmed by grep across all door-Pi services (see trust boundaries).
