Doorboard — how it works at runtime

A smart doorbell split into a real-time door plane (hallway, latency-critical, works offline) and a control plane (in-room, latency-tolerant). The rule that shapes everything: the door interaction must complete on local hardware alone — the NUC, NAS, and internet can all be down and the doorbell still works.

Critical path — must be instant, local only Non-critical — may queue / lag / retry High trust Medium/low trust (stealable) Low trust (no secrets)

1 The devices & where they live

Two planes joined by a single Ethernet link + protected power loop.

🚪 Real-time door plane — hallway, medium/low trust, latency-critical

Removable door assembly. Physically reachable, so it holds no admin secrets and no long-term archive.

ESP32-S3 no secrets

The instant-feedback microcontroller
  • Button debounce + knock sensor
  • LED ring + speaker: generic feedback <30 ms
  • Profile cache = opaque IDs only
  • Watchdog fallback if Pi is silent

Raspberry Pi 5 per-device tokens

+ Hailo-8 AI HAT, USB SSD. Runs the door services:
  • door-visiond — Hailo, cameras, identity cache
  • door-media — MediaMTX, recording
  • door-api — session state machine
  • door-ui — kiosk SPA (5 routes)
  • door-sync — upload queue → NUC/NAS

Displays public

Driven by the Pi over HDMI + WebSocket
  • Wallboard (21–24"): ambient tiles, visitor mode
  • DoorPad (7" touch): visitor interaction
  • Broad status only — never private data

Cameras on-Pi

  • Recognition cam (standard) → visiond
  • Visitor cam (wide NoIR) → media
  • Stream is always live — no cold start on press
▲   single Ethernet + protected power   (non-critical link — door works if it's cut)   ▼

🏠 Control plane — inside room, trusted, latency-tolerant

Holds the admin secrets and durable history. Never sits in the door interaction path.

NUC high trust

The only holder of admin secrets
  • control-plane-api + PostgreSQL (history)
  • wallboard-worker — ambient jobs
  • Mosquitto (MQTT), Home Assistant, Caddy

BirdNET Pi 4 isolated

  • BirdNET-Go acoustic ID (ambient tile source)
  • Optional SDR/ADS-B Pi for aircraft

NAS high trust

  • Durable media archive only
  • Never an active DB, never realtime
  • Pi uploads via a limited service credential

5 When something is down

The plane split exists precisely so failures degrade gracefully instead of breaking the door.

✅ Still fully works with NUC / NAS / internet down

  • Button → generic feedback (<30 ms) — pure ESP32
  • Cached-profile personalized greeting (<100 ms)
  • Session, visitor mode, DoorPad interaction (door-api, local)
  • Live video + local recording to SSD
  • ESP32 watchdog fallback if even the Pi goes silent

⏳ Degrades / queues (catches up on recovery)

  • Event history & Home Assistant notifications (queued in door-sync)
  • NAS archival — clips wait on the SSD, upload later
  • Ambient tiles go stale until the NUC/sources return
  • New face enrollment & admin actions (need the control plane)
