# packages/auth — tokens and admin authentication

Security-sensitive: changes here always get Claude-tier review (ADR-0005/0008).

- **Service tokens (implemented, T-501):** `doorboard_auth.tokens` — issuance, verification, and revocation of limited-scope Pi tokens (`ingest`, `upload`, `config` scopes). Pure logic, no I/O: a token is `dbt_<token_id>_<secret>`; only `sha256(secret)` is ever persisted; lookup is by the public `token_id`, comparison is constant-time (`hmac.compare_digest`). Consuming services (currently `control-plane-api`) own the actual `ServiceTokenRecord` storage — revocation takes effect on the next request because verification always re-reads the store, never caches a decision. See `apps/control-plane-api/src/control_plane_api/tokens.py` for the Postgres-backed store and CLI, and its README for the rotation runbook.
- **Admin auth:** session-based login for `/admin` routes on the Pi and NUC (single-owner model; simple credential + session cookie, rate-limited, no public signup) is **not built yet**. `door-api`'s social moderation panel and `control-plane-api`'s admin/moderation endpoints use a stopgap shared bearer secret (`*_ADMIN_TOKEN` env var, `secrets.compare_digest`) documented in each service's own module. Replace both call sites with a real session system here when that work is scheduled — don't duplicate a third stopgap.
- **Visitor tokens:** short-lived signed tokens embedded in QR codes for `/visitor` flows; scoped to one session, minutes-long TTL, rate-limited verification. Not built yet.
- **Rules:** no secrets in git; tokens never logged; constant-time comparisons; the door Pi can *hold* only its own limited tokens (trust model §2). Mutual TLS is a future ADR if warranted — don't build speculative PKI.
